> **⛔ CRITICAL (2026-06-26): live refresh verification was REMOVED — the temp-table probe leaks into Desktop and crashes it. DO NOT re-enable `MQueryEvaluator.evaluate_m` in any UI path that runs against an open Desktop.** Proven on the user's real model via a frown report: adding the probe table `__pbi_ai_mq_probe` via TOM makes **Power BI Desktop sync it into its own Mashup document (`Section1`)**; our TOM cleanup removes it from the AS engine (confirmed: `$SYSTEM.TMSCHEMA_TABLES` no longer has it) **but CANNOT remove it from Desktop's Mashup document** — so a ghost `__pbi_ai_mq_probe` query persists in the Power Query navigator and crashes Desktop (`NullReferenceException` in `QueriesNavigatorModelBase.IsQueryGroupNode`, same signature as the M write-back) the moment the user clicks/double-clicks a query. So **any TOM model mutation (write-back OR the verification probe) is unsafe on an open Desktop.** The M assistant is now **read-only**: grounded generation + static lint (`MScriptArtifact.validate`, `run_verified=False`) only; the user pastes into the Advanced Editor and Desktop verifies on Apply. **DAX is unaffected** — DAX verification uses `EVALUATE` (a read-only query, creates no table), so it never leaks/crashes; only the M temp-table approach does. Cleanup for a leaked probe: delete `__pbi_ai_mq_probe` in Power Query editor via right-click→删除 (avoid double-click), then 关闭并应用.

The original (now-disabled) mechanism, kept for the record — **do not resurrect against a live Desktop**:

Phase-2 needs the same "run-verify, never assert" guarantee as phase-1 DAX, but **M has no standalone queryable endpoint** — you cannot send an M expression to the engine and get a result the way `EVALUATE` runs DAX. The only faithful verification is a **refresh round-trip** (worked in standalone tests, fatal with Desktop open). Mechanism (in `mquery/live_eval.py` `MQueryEvaluator.evaluate_m`, now uncalled):

1. Via TOM, add a **uniquely-named temp table** `__pbi_ai_mq_probe` whose `MPartitionSource.Expression` wraps the candidate M.
2. `table.RequestRefresh(RefreshType.Full)` then `model.SaveChanges()` — **this runs the real Mashup engine against the real data source.** Credentials are inherited (no prompt) when the M references an existing query/source the open model already has.
3. Read back the result via ADOMD `EVALUATE`; then remove the temp table + `SaveChanges()` in a `finally` (never touches the user's real tables).

**Two non-obvious gotchas that shaped the design:**
- **A raw M (import) partition added via TOM has NO inferred columns.** Power BI Desktop normally runs schema detection when you create a query; doing it externally via TOM does not. So a naked `EVALUATE '__pbi_ai_mq_probe'` fails with `表"…"不能在计算中使用，因为它没有任何列`. **Fix:** make the probe return `Table.Schema(candidate)` — whose output shape is FIXED and known up front (columns Name/TypeName/Kind/...). Declare those fixed columns on the probe table, refresh, and `EVALUATE` returns the *candidate's output schema as data rows* (column names + M types). EVALUATE row dict keys come back prefixed: `'__pbi_ai_mq_probe[Name]'`.
- **A broken candidate throws on `SaveChanges()` (after RequestRefresh), not on EVALUATE.** The exception is an `OperationException` whose text wraps the real Mashup error, e.g. `...Error returned: 'OLE DB or ODBC error: [Expression.Error] 找不到表的"X"列。'`. `_clean_m_error` extracts the `[Expression.Error]/[DataFormat.Error]/[DataSource.Error]` core — that's exactly what the repair loop feeds back to the LLM. `run_verified=True` on both success and verified failure.

**Write-back of M is NOT possible on an open Desktop model — removed 2026-06-25 after a real crash.** Setting an existing query's `MPartitionSource.Expression` via TOM + `SaveChanges()` makes **Power BI Desktop itself crash**: `System.NullReferenceException` in `Microsoft.Mashup.Client.UI...Ux.Navigator.QueriesNavigatorModelBase.IsQueryGroupNode` (frown report confirmed, Store version 2.155.756.0). Editing Power Query (the Mashup document) externally on an OPEN Desktop is unsupported — Desktop keeps the authoritative mashup doc and its query navigator throws when an external XMLA write touches a partition's M. **Crucial contrast that the stack trace proves: measure / calculated-table write-back ([[write-back-via-tom]]) is SAFE — those are pure model objects that never touch the Mashup UI, so they cannot hit this bug.** So `write_m_partition` was removed; the M assistant shows the run-verified M with a copy button + guidance to paste into Power Query's **高级编辑器 (Advanced Editor)** (主页→转换数据→选中查询→高级编辑器→粘贴→完成→关闭并应用). The model is not corrupted by the failed attempt (the rejected `SaveChanges` doesn't persist).

A data **sample** (actual rows, not just schema) is a deliberate non-goal of v1: it would need declaring all output columns dynamically (a second refresh) or a JSON-serialize trick that can false-negative on some types. Schema + "refresh succeeded" is the robust honest verification. See [[dev-progress-2026-06]], [[live-evaluate-validation-is-required]], [[dax-and-engine-gotchas]].
