**When packaging for distribution, ALWAYS ASK the user which format first: (a) zip (extract-and-run) or (b) exe (Inno Setup installer).** Don't assume — they want to choose each time.
**Why:** the two suit different situations (quick share vs. "install software" experience) and the user wants control over which one is produced.
**How to apply:** at the start of any packaging request, ask zip vs exe; then build that one with the recipe below. Both share the same underlying zero-prereq bundle.

- **(b) exe — Inno Setup installer** → `PowerBI助手_安装.exe` (~96 MB; lzma2 beats zip). Per-user, no admin (`DefaultDirName={localappdata}\PowerBI助手`, `PrivilegesRequired=lowest`). Script: repo `packaging/installer.iss` (**must be UTF-8 WITH BOM** or Chinese compiles garbled on a GBK Windows). Compiler via `winget install --id JRSoftware.InnoSetup` (ISCC at `%LOCALAPPDATA%\Programs\Inno Setup 6\ISCC.exe`). Creates Start-Menu/Desktop shortcuts + uninstaller. Avoid PyInstaller `--onefile` (Streamlit + pythonnet make it very painful).
- **(a) zip** → `PowerBI助手.zip` (~161 MB), extract-and-run.

**`.vbs` double-click is unreliable — launch it via `wscript.exe` explicitly.** Many corporate machines remap the `.vbs` association to Notepad (security hardening), so double-clicking `启动.vbs` just opens it in an editor. The WSH *engine* is still fine (`cscript`/`wscript file.vbs` run). So: the installer's Start-Menu/Desktop shortcuts target `{sys}\wscript.exe` with the `.vbs` as a parameter (`Filename: "{sys}\wscript.exe"; Parameters: """{app}\packaging\启动.vbs"""`) — this gives the hidden-console + app-mode window regardless of the association. For the zip (no shortcuts) tell users to double-click `启动.bat` (console shows) and NOT `启动.vbs`. The launcher itself: `启动.vbs` runs the sibling `启动.bat` hidden; `启动.bat` starts Streamlit headless, waits for the port, then opens an **Edge app-mode window** (`msedge --app=http://localhost:8501`) and stops the server when that window closes. Use a **unique `--user-data-dir` per launch** so `start /wait` actually blocks until the window closes (a shared profile hands off to an existing Edge process and returns immediately → the bat would kill the server right after opening it).

The app is a **Streamlit web app** (local server + browser UI) and connects to the user's **local** Power BI Desktop engine, so it **must run on each user's own machine — it cannot be hosted on a server** for shared access. Distribution = a per-machine local install. Prereqs for each colleague: Windows + Power BI Desktop installed + their own LLM API key (key is stored per-user under `~/.powerbi_ai_assistant`).

**Working recipe (verified, produces a ~161 MB zero-prereq zip):** bundle a relocatable Python (`astral-sh/python-build-standalone`, the `*-x86_64-pc-windows-msvc-install_only.tar.gz` asset — extracts to a self-contained `python/` folder, rename to `runtime/`), `pip install -r requirements-runtime.txt` into it, lay out `runtime/ + powerbi_ai_assistant/ + .streamlit/ + packaging/`, zip it. A double-click `启动.bat` launcher picks Python by priority **bundled `runtime\` > local `.venv\` > system python** and runs `python -m streamlit run ...`. Source lives in repo `packaging/` (`启动.bat`, `requirements-runtime.txt`, `使用说明.md`, `打包说明.md`). Runtime deps exclude dev tools (pytest/ruff/black/mypy) and the dropped `pbixray`.

**Pitfalls hit and fixed (all verified by real runs):**
1. **The `.bat` must be pure ASCII** — no BOM, no Chinese, no em-dash. A UTF-8 `.bat` with non-ASCII bytes is mis-parsed by the GBK Windows shell: it corrupts whole lines (e.g. `cd /d` silently fails), so it falls through to the wrong Python. Symptom: `'EnableExtensions' / '/d' is not recognized`, and the listener ends up being system python instead of `runtime\`. Put all Chinese in the README, not the launcher. Verify with `grep -nP '[^\x00-\x7F]' 启动.bat`.
2. **Streamlit's first-run "enter your email" prompt blocks startup** on a fresh machine (hangs, never binds the port). Fix: before launching, write `%USERPROFILE%\.streamlit\credentials.toml` with `[general]` + `email = ""` (or pass `--server.headless true`, which also skips it but then you must open the browser yourself).
3. **Windows MAX_PATH (260 chars)** — the `anthropic` package ships 80+ char filenames (e.g. `beta_managed_agents_self_hosted_resources_unsupported_deployment_paused_reason_error.py`). Building/extracting under a long base path → `OSError: [Errno 2] No such file or directory` during pip install / unzip. **Build, zip, AND extract under a SHORT path** (e.g. `C:\PowerBI助手`). The scratchpad path is too long — build in `C:\pbibuild`. Tell colleagues to extract to `C:\`, not a deep/Desktop/OneDrive folder.
4. **Behind the corporate MITM proxy** ([[corporate-tls-inspection-needs-truststore]]): `pip install` needs `--trusted-host pypi.org --trusted-host files.pythonhosted.org`; to download the portable Python use **Windows' built-in curl** (`/c/Windows/System32/curl.exe`, uses Schannel/OS trust store) — Git Bash's own `curl` uses a private CA bundle and fails.

**Verification done / not done:** verified end-to-end by extracting the zip to a fresh short path and confirming `启动.bat` starts the server on the bundled `runtime\python.exe` (checked the listener PID's exe path). NOT verified on a truly clean machine (no Python / no internet / no Power BI) — do that before wide rollout. Also: first **write-back** still downloads TOM (~30 MB) from NuGet once (read/generate/validate don't).
